# CredVault Database Architecture Documentation

## Overview

**Technology Stack:** Microsoft SQL Server + Entity Framework Core 10.0  
**Architecture Pattern:** Clean Architecture / Microservices (Database-per-Service)  
**Message Broker:** MassTransit with RabbitMQ (for cross-service communication)

---

## Cross-Service Relationships

Since each service has its own database, **cross-service relationships are handled via GUID IDs without foreign key constraints**.

```
┌─────────────────┐     ┌─────────────────┐
│  Identity       │     │  Card           │
│  Service        │     │  Service        │
│                 │     │                 │
│  identity_users │─────┼── UserId        │
│                 │     │                 │
└─────────────────┘     └────────┬────────┘
                                │
         ┌──────────────────────┼──────────────────────┐
         │                      │                      │
         ▼                      ▼                      ▼
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  Card           │     │  Billing        │     │  Payment        │
│  Service        │     │  Service        │     │  Service        │
│                 │     │                 │     │                 │
│  CreditCards    │─────┼── CardId        │─────┼── CardId        │
│                 │     │                 │     │                 │
│  CardIssuers    │     │  Bills          │     │  Payments       │
│                 │     │                 │     │                 │
│  CardTransac-   │     │  Statements     │     │  Transactions   │
│  tions          │     │                 │     │                 │
│                 │     │  RewardAccounts │     │  Sagas          │
```

## Data Flow Examples

### 1. Add New Card
```
Identity (User) → Card Service (CreditCard with UserId)
                → CardIssuers (if new issuer)
```

### 2. Generate Bill
```
Card Service (CardTransactions) → Billing Service (Bill)
                                → Billing Service (Statement)
                                → Billing Service (StatementTransactions)
```

### 3. Process Payment
```
Payment Service: Payment created
    ↓
Card Service: Card verified, balance checked
    ↓
Billing Service: Bill status updated, rewards calculated
    ↓
Payment Service: Transaction logged, saga completed
```

### 4. Reward Redemption
```
Payment triggers → RewardTransactions (Earned)
                → RewardAccounts (PointsBalance updated)
```

## Database Patterns

| Pattern | Implementation |
|---------|----------------|
| Audit Fields | All tables have `CreatedAtUtc`, `UpdatedAtUtc` |
| Soft Delete | CardService uses `IsDeleted` with query filters |
| Indexing | Strategic indexes on FKs and queried columns |
| GUID Primary Keys | All primary keys are GUID |
| Decimal Precision | Money values use `decimal(18,2)` |
| Cascading Deletes | Within service boundaries |
