# SAGA PATTERN - COMPLETE LEARNING GUIDE (RABBITMQ + EVENTS + ROLLBACK)

**Context: CredVault payment flow across microservices**

---

## Why You Need Saga Here

In this system, one payment touches multiple services:

1. Payment Service (create and orchestrate payment)
2. Billing Service (mark bill paid)
3. Billing Service (redeem rewards)
4. Card Service (deduct card outstanding balance)
5. Notification Service (send success/failure + OTP notifications)

If one step fails after previous steps succeeded, a normal DB transaction cannot rollback across service boundaries.

That is exactly why Saga is used.

Saga = distributed transaction coordination using events + compensating actions.

## RabbitMQ + MassTransit Setup (How Messages Move)

All services use MassTransit with RabbitMQ.

What matters most:

1. Consumers are attached to explicit queue endpoints.
2. Services bind to message contracts they care about.
3. Retry policy is enabled on key endpoints.
4. In-memory outbox is enabled to reduce duplicate publish side effects.

Common retry setup used in service queues:
- 1 second
- 5 seconds
- 15 seconds

Examples in service Program files:

- Payment service queues:
  - `payment-orchestration` (saga state machine)
  - `payment-process` (processing/compensation workers)
  - `payment-domain-event` (final payment state consumers)

- Billing service queue:
  - `billing-domain-event` binds `IBillUpdateRequested` and `IRevertBillUpdateRequested`

- Card service queue:
  - `card-domain-event` binds `ICardDeductionRequested` and `IPaymentReversed`

- Notification service queue:
  - `notification-domain-event` consumes `IPaymentOtpGenerated`, `IPaymentCompleted`, `IPaymentFailed`, etc.

## The Most Important Concept: CorrelationId

Every saga message carries `CorrelationId`.

In this implementation:

- `CorrelationId` is set to payment id when saga starts.
- State machine correlates all events by this id (`CorrelateById`).
- You can trace one payment across all services by this single value in logs.

Without `CorrelationId`, distributed debugging becomes painful.

## Saga State Machine (Mental Model)

States used by `PaymentOrchestrationSaga`:

1. `Initial`
2. `AwaitingOtpVerification`
3. `AwaitingPaymentConfirmation`
4. `AwaitingBillUpdate`
5. `AwaitingRewardRedemption`
6. `AwaitingCardDeduction`
7. `Compensating`
8. `Compensated`
9. `Completed`
10. `Failed`

The saga state row stores progress flags like:

- `OtpVerified`
- `PaymentProcessed`
- `BillUpdated`
- `RewardsRedeemed`
- `CardDeducted`
- `CompensationAttempts`
- detailed error fields for failure reason tracking

## Event-Driven Happy Path (Step By Step)

### Step 1: Payment initiation

`InitiatePaymentCommand`:

1. Creates payment with `Status=Initiated`
2. Generates OTP
3. Publishes `IStartPaymentOrchestration`
4. Publishes `IPaymentOtpGenerated`

### Step 2: OTP verification

`VerifyOtpCommand` sends to queue `payment-orchestration`:

- success: `IOtpVerified`
- failure: `IOtpFailed`

Saga transition:
- `AwaitingOtpVerification` -> `AwaitingPaymentConfirmation` on `IOtpVerified`

Then saga publishes:
- `IPaymentProcessRequested`

### Step 3: Payment processing worker

`PaymentProcessConsumer`:

1. Finds payment
2. Marks status `Processing`
3. Publishes `IPaymentProcessSucceeded` (or `IPaymentProcessFailed`)

Saga transition on success:
- `AwaitingPaymentConfirmation` -> `AwaitingBillUpdate`

Then saga publishes:
- `IBillUpdateRequested`

### Step 4: Bill update in Billing Service

`BillUpdateSagaConsumer`:

1. Executes `MarkBillPaidCommand`
2. On success publishes `IBillUpdateSucceeded`
3. On error publishes `IBillUpdateFailed`

Saga on bill success:

- if rewards amount > 0:
  - transition to `AwaitingRewardRedemption`
  - publish `IRewardRedemptionRequested`

- if rewards amount <= 0:
  - transition to `AwaitingCardDeduction`
  - publish `ICardDeductionRequested`

### Step 5: Reward redemption (optional)

`RewardRedemptionConsumer` (Payment service side worker):

1. Calls Billing internal redeem endpoint
2. Publishes `IRewardRedemptionSucceeded` or `IRewardRedemptionFailed`

On success saga moves to card deduction and publishes `ICardDeductionRequested`.

### Step 6: Card deduction in Card Service

`CardDeductionSagaConsumer`:

1. Idempotency guard using existing transaction description `Saga:{CorrelationId}`
2. Deducts amount from card outstanding balance
3. Writes a card transaction
4. Publishes `ICardDeductionSucceeded` (or `ICardDeductionFailed`)

Saga on card success:

- transitions to `Completed`
- publishes `IPaymentCompleted`

### Step 7: Completion side effects

`PaymentCompletedConsumer`:

1. Marks payment `Completed`
2. Adds transaction ledger record
3. idempotent check if already completed

`NotificationService.DomainEventConsumer` handles `IPaymentCompleted` for user notification.

## Rollback / Compensation Flow (The Main Part You Asked)

Compensation starts when saga enters `Compensating`.

This implementation uses explicit compensating events, not implicit DB rollback.

### Failure Case A: Bill update fails

Trigger:
- saga gets `IBillUpdateFailed`

Action:
- saga publishes `IRevertPaymentRequested`

Consumer:
- `RevertPaymentConsumer`

What `RevertPaymentConsumer` does:

1. Finds payment
2. If already reversed -> publishes `IRevertPaymentSucceeded` (idempotent behavior)
3. Sets payment status to `Reversed`
4. Inserts reversal ledger transaction
5. Publishes `IPaymentReversed`
6. Publishes `IRevertPaymentSucceeded`

Downstream effect:
- Card service `PaymentReversedConsumer` receives `IPaymentReversed` and restores card balance.

Saga finish for this case:
- On `IRevertPaymentSucceeded`: transition to `Compensated`, then publish `IPaymentFailed`

---

### Failure Case B: Reward redemption fails

Trigger:
- saga gets `IRewardRedemptionFailed`

Action:
- saga publishes `IRevertPaymentRequested`

Then same compensation path as Case A.

Reasoning:
- If rewards redemption fails, system prefers consistency and compensates the payment chain.

---

### Failure Case C: Card deduction fails

Trigger:
- saga gets `ICardDeductionFailed`

Action sequence:

1. Saga -> `Compensating`
2. Saga publishes `IRevertBillUpdateRequested`
3. Billing `RevertBillSagaConsumer` reverts bill update and publishes:
   - `IRevertBillUpdateSucceeded` OR
   - `IRevertBillUpdateFailed`

4. If bill revert succeeded:
   - saga publishes `IRevertPaymentRequested`
   - payment revert path executes (same as Case A)

5. If bill revert fails:
   - saga increments `CompensationAttempts`
   - retries compensation
   - after threshold, transitions to `Failed`

Compensation retry thresholds in current saga:

- bill revert failures: fail permanently after 3 attempts
- payment revert failures: fail permanently after 5 attempts

## What "Rollback Everything" Means In This System

Rollback is logical, not one giant SQL transaction.

It means:

1. Revert payment aggregate and ledger (`IRevertPaymentRequested`)
2. Revert bill paid state (`IRevertBillUpdateRequested`) when needed
3. Restore card balance (`IPaymentReversed` consumed by card service)
4. Emit terminal failure event (`IPaymentFailed`)

Each service undoes only its own data.

That is the core microservice rule.

## Idempotency + Reliability Safeguards You Already Have

### 1. Idempotent consumers

- Card deduction consumer checks if saga transaction already exists.
- Payment completed consumer exits if payment already completed.
- Revert payment consumer exits with success if already reversed.

### 2. Message retries

- queue retries configured with intervals 1s, 5s, 15s.

### 3. InMemoryOutbox

- reduces duplicate publish side effects during consumer execution.

### 4. Durable saga state

- saga state stored in SQL table `PaymentOrchestrationSagas`.

## How To Debug Saga Fast

When something breaks, trace by `CorrelationId` in this order:

1. Payment Service logs (saga transitions)
2. Billing Service logs (bill update/revert)
3. Card Service logs (card deduction/payment reversed)
4. Notification Service logs (final user-facing event)

Useful checkpoints:

- Did `IStartPaymentOrchestration` publish?
- Did queue `payment-orchestration` receive OTP event?
- Which terminal event happened: `IPaymentCompleted` or `IPaymentFailed`?
- Did compensation events publish and get consumed?

"Orchestrate -> Confirm each local step with events -> If any step fails, publish compensating events in reverse business order until system is consistent again."
