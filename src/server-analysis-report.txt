## EXECUTIVE SUMMARY

The CredVault server is a microservices-based backend application built with .NET 10,
designed to manage credit card services, billing, payments, and notifications.
The system follows a microservices architecture with 5 distinct services communicating
via RabbitMQ message broker and an API Gateway using Ocelot.

### Architecture Overview
- API Gateway: Ocelot-based reverse proxy (Port 5006)
- Card Service: Credit card management (Port 5002)
- Payment Service: Payment orchestration with SAGA pattern (Port 5004)
- Billing Service: Bills, statements, rewards management (Port 5003)
- Notification Service: Email notifications (Port 5005)
- Shared Contracts: Common DTOs, events, and base classes

### SAGA Orchestration State Machine:

States:
- Initial
- AwaitingOtpVerification
- AwaitingPaymentConfirmation
- AwaitingBillUpdate
- AwaitingCardDeduction
- Completed (Final)
- Compensating
- Compensated (Final)
- Failed (Final)

Events:
- StartOrchestration
- OtpVerified / OtpFailed
- PaymentSucceeded / PaymentFailed
- BillUpdateSucceeded / BillUpdateFailed
- CardDeductionSucceeded / CardDeductionFailed
- RevertBillSucceeded / RevertBillFailed
- RevertPaymentSucceeded / RevertPaymentFailed

Flow:
1. StartOrchestration → AwaitingOtpVerification
2. OtpVerified → AwaitingPaymentConfirmation (publishes IPaymentProcessRequested)
3. PaymentSucceeded → AwaitingBillUpdate (publishes IBillUpdateRequested)
4. BillUpdateSucceeded → AwaitingCardDeduction (publishes ICardDeductionRequested)
5. CardDeductionSucceeded → Completed (publishes IPaymentCompleted)

Compensation Flow:
- If BillUpdateFailed → Compensating → RevertPayment
- If CardDeductionFailed → Compensating → RevertBillUpdate → RevertPayment
- Max 3 compensation attempts for bill revert
- Max 5 compensation attempts for payment revert

## 8. DATA FLOW ANALYSIS

### Card Creation Flow:
1. POST /api/v1/cards with card details
2. Validates issuer, detects network, checks duplicates
3. Creates CreditCard in CardService DB
4. Publishes ICardAdded event
5. NotificationService consumes and sends email
6. Returns created card

### Payment Flow (Full SAGA):
1. POST /api/v1/payments/initiate
2. PaymentService validates bill with BillingService
3. Generates OTP, stores in Payment record
4. Publishes IStartPaymentOrchestration, IPaymentOtpGenerated
5. SAGA enters AwaitingOtpVerification state
6. User submits OTP via POST /verify-otp
7. PaymentService publishes IOtpVerified
8. SAGA transitions to AwaitingPaymentConfirmation
9. PaymentProcessConsumer receives IPaymentProcessRequested
10. Updates payment status, publishes IPaymentProcessSucceeded
11. SAGA transitions to AwaitingBillUpdate
12. BillUpdateSagaConsumer receives IBillUpdateRequested
13. BillingService marks bill paid, publishes IBillUpdateSucceeded
14. SAGA transitions to AwaitingCardDeduction
15. CardDeductionSagaConsumer receives ICardDeductionRequested
16. CardService deducts from outstanding balance, publishes ICardDeductionSucceeded
17. SAGA transitions to Completed
18. Publishes IPaymentCompleted
19. NotificationService sends payment confirmation

### Overdue Bill Flow:
1. Admin calls POST /api/v1/billing/bills/admin/check-overdue
2. BillingService identifies overdue bills
3. Publishes IBillOverdueDetected for each
4. CardService BillOverdueConsumer receives event
5. Applies strike to card via ApplyStrikeCommand
6. If 3 strikes reached, card blocked
7. Publishes ICardBlocked event
8. NotificationService sends warning

### Refund/Reversal Flow:
1. Payment fails at any SAGA step
2. SAGA enters Compensating state
3. Reverts in reverse order:
   - RevertBillUpdateRequested → IRevertBillUpdateSucceeded/Failed
   - RevertPaymentRequested → IPaymentReversed + IRevertPaymentSucceeded/Failed
4. NotificationService notified of failure
